# Техническое задание: HTTP/HTTPS debug proxy

## 1. Цель

Разработать сервис на Python для Ubuntu, который принимает входящие HTTP или HTTPS запросы, полностью логирует их, пересылает на целевой сервер, получает ответ и возвращает этот ответ исходному клиенту.

## 2. Назначение

Сервис используется как отладочный прокси между клиентом и целевым сервером.

Основные задачи:

- принять запрос от клиента;
- полностью сохранить входящий запрос;
- переслать запрос на целевой сервер;
- получить ответ целевого сервера;
- полностью сохранить ответ целевого сервера;
- вернуть ответ целевого сервера исходному клиенту.

## 3. Режимы работы

Сервис должен поддерживать два режима приема:

- `HTTP`;
- `HTTPS`.

Один запущенный экземпляр сервиса работает только в одном режиме.

Порт по умолчанию:

- `HTTP`: `51400`;
- `HTTPS`: `51401`.

Режим работы, порт прослушивания и параметры TLS должны задаваться через `ini`-файл.

## 4. Конфигурация

Все основные параметры должны задаваться во внешнем `ini`-файле, лежащем рядом со скриптом запуска или переданном через параметр командной строки.

В `ini` должны быть предусмотрены настройки:

- режим приема: `http` или `https`;
- адрес прослушивания;
- порт прослушивания;
- адрес целевого сервера;
- порт целевого сервера;
- схема пересылки на целевой сервер: `http` или `https`;
- таймаут подключения к целевому серверу;
- таймаут ожидания ответа от целевого сервера;
- каталог логов;
- путь к TLS-сертификату сервиса для режима `https`;
- путь к приватному ключу сервиса для режима `https`.

Проверка TLS-сертификата целевого HTTPS-сервера не требуется.

## 5. Прием входящих запросов

Сервис должен:

- принимать запросы на любых URL-путях;
- принимать любые query string;
- поддерживать любые HTTP-методы, включая `GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `HEAD`, `OPTIONS` и нестандартные методы;
- принимать любые заголовки;
- принимать body запроса полностью;
- читать request body полностью в память для последующего логирования и пересылки.

## 6. Пересылка запроса

Сервис должен пересылать запрос на целевой сервер, указанный в `ini`.

При пересылке необходимо:

- сохранить исходный HTTP-метод;
- сохранить исходный path;
- сохранить исходный query string;
- сохранить исходные заголовки, включая оригинальный `Host`;
- сохранить body запроса без изменений;
- использовать схему, адрес и порт целевого сервера из `ini`.

Пример:

Если клиент отправил:

```text
POST /api/v1/codes/check?x=1
```

а в `ini` указан целевой сервер:

```text
target_scheme = https
target_host = 192.168.3.20
target_port = 443
```

то прокси должен переслать запрос на:

```text
https://192.168.3.20:443/api/v1/codes/check?x=1
```

## 7. Обработка ответа целевого сервера

Сервис должен:

- получить HTTP-статус ответа целевого сервера;
- получить все заголовки ответа целевого сервера;
- получить body ответа целевого сервера полностью;
- вернуть исходному клиенту статус, заголовки и body ответа целевого сервера.

Body ответа целевого сервера нужно читать полностью в память для логирования.

## 8. Логирование

Для каждого входящего запроса должны создаваться отдельные файлы:

- один `txt`;
- один `json`.

Имена файлов должны строиться по шаблону:

```text
timestamp + request_id
```

Пример:

```text
2026-05-26T12-34-56.123456Z_9f2e6c7a.txt
2026-05-26T12-34-56.123456Z_9f2e6c7a.json
```

Для одного запроса `txt` и `json` должны иметь одинаковую базовую часть имени.

### 8.1. Что логировать по входящему запросу

Необходимо сохранять:

- дату и время получения;
- `request_id`;
- режим приема: `http` или `https`;
- IP-адрес и порт клиента;
- HTTP-метод;
- полный path;
- query string;
- HTTP-версию;
- все входящие заголовки;
- body входящего запроса полностью;
- размер body;
- результат декодирования body в читаемый вид, если возможно.

### 8.2. Что логировать по ответу целевого сервера

Необходимо сохранять:

- URL, на который был переслан запрос;
- схему, host и порт целевого сервера;
- HTTP-статус ответа;
- все заголовки ответа;
- body ответа полностью;
- размер body ответа;
- результат декодирования body ответа в читаемый вид, если возможно;
- время выполнения запроса к целевому серверу.

### 8.3. Ошибки пересылки

Если пересылка на целевой сервер завершилась ошибкой, в лог нужно записать:

- тип ошибки;
- текст ошибки;
- URL целевого сервера;
- момент возникновения ошибки;
- данные входящего запроса.

## 9. Форматы логов

### 9.1. TXT

TXT-файл должен быть удобен для ручного чтения и содержать:

- сведения о входящем запросе;
- сведения о пересылке;
- сведения об ответе целевого сервера;
- body входящего запроса;
- body ответа целевого сервера;
- ошибки пересылки, если они возникли.

### 9.2. JSON

JSON-файл должен содержать один JSON-объект со всей информацией по запросу, пересылке и ответу.

## 10. Обработка ошибок

Если целевой сервер вернул HTTP-ответ, прокси должен вернуть исходному клиенту именно этот ответ, включая статус.

Если запрос не удалось переслать или ответ не удалось получить из-за сетевой ошибки, таймаута, TLS-ошибки или иной ошибки проксирования, прокси должен вернуть клиенту:

```text
502 Bad Gateway
```

Тело ответа при ошибке должно быть JSON с описанием ошибки.

## 11. TLS

Для режима приема `https` сервис должен использовать:

- TLS-сертификат из `ini`;
- приватный ключ из `ini`.

Проверка TLS-сертификата целевого сервера при пересылке не требуется.

## 12. Нефункциональные требования

- Сервис должен работать на Ubuntu.
- Реализация должна быть простой для запуска из консоли.
- Не требуется streaming больших файлов.
- Допускается чтение request body и response body полностью в память.
- Логи должны быть пригодны для ручного чтения и машинной обработки.
- Конфигурация должна меняться без изменения исходного кода.

## 13. Критерии приемки

Сервис считается готовым, если:

- запускается на Ubuntu из консоли;
- читает настройки из `ini`;
- в режиме `http` слушает порт `51400` по умолчанию;
- в режиме `https` слушает порт `51401` по умолчанию;
- принимает любые HTTP-методы и пути;
- пересылает запросы на целевой сервер с сохранением метода, path, query string, заголовков и body;
- сохраняет оригинальный заголовок `Host` клиента при пересылке;
- возвращает клиенту ответ целевого сервера;
- логирует входящий запрос и ответ целевого сервера в отдельные `txt` и `json` файлы на каждый запрос;
- логирует body запроса и body ответа полностью;
- при ошибке пересылки возвращает `502 Bad Gateway`;
- проверка TLS-сертификата целевого HTTPS-сервера не выполняется.

## 14. Ожидаемый результат

В результате должны быть подготовлены:

- исходный код Python-сервиса;
- пример `ini`-файла;
- инструкция по запуску в режиме `http`;
- инструкция по запуску в режиме `https`;
- описание формата логов;
- пример проверки через `curl`.
