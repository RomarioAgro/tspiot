# Техническое задание

## 1. Цель

Разработать отладочный сервис на Python, принимающий HTTPS-запросы на порту `51401` и сохраняющий каждый запрос полностью в отдельных лог-файлах `txt` и `json`.

## 2. Назначение сервиса

Сервис используется как универсальный приемник входящих HTTPS-запросов для отладки интеграций. Основная задача сервиса: принять запрос, сохранить его в максимально полном и удобном для анализа виде и вернуть клиенту успешный ответ.

## 3. Область применения

- Сервер размещается на `Ubuntu`.
- Клиент, отправляющий запросы, работает на `Windows`.
- Клиент должен доверять TLS-сертификату сервера.
- Сервис запускается вручную из консоли, без обязательного оформления в `systemd`.

## 4. Функциональные требования

### 4.1. Сетевое поведение

- Протокол: `HTTPS`.
- Порт прослушивания: `51401`.
- Сервис должен принимать запросы на любых URL-путях.
- Сервис должен принимать любые HTTP-методы, включая `GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `HEAD`, `OPTIONS` и другие нестандартные методы, если они придут от клиента.
- Сервис должен принимать подключения от любого клиента, имеющего сетевой доступ к порту `51401`.
- Достаточно однопоточной обработки запросов.

### 4.2. Ответ сервиса

- На любой корректно принятый HTTP-запрос сервис должен возвращать `200 OK`.
- Возврат `200 OK` не должен зависеть от HTTP-метода, URL-пути, наличия body или результата декодирования body.

### 4.3. Поддержка путей

Сервис не должен содержать фиксированного набора endpoint-ов.

Он должен принимать, логировать и обслуживать любые пути, например:
- `/`
- `/api/test`
- `/debug/incoming`
- `/anything/else`

### 4.4. Данные, которые нужно логировать

Для каждого входящего запроса необходимо сохранить:
- дату и время получения запроса;
- уникальный идентификатор запроса `request_id`;
- HTTP-метод;
- полный URL-путь;
- query string;
- HTTP-версию;
- все заголовки без маскирования и без удаления значений;
- body запроса в полном объеме;
- размер body;
- `Content-Type`, если присутствует;
- IP-адрес клиента;
- порт клиента;
- результат декодирования body в читабельный вид, если это возможно;
- дополнительную служебную информацию, если она может быть извлечена без потери данных.

### 4.5. Требования к body

- Body нужно логировать полностью, без ограничения размера и без усечения.
- Маскирование чувствительных данных не требуется.
- Заголовки `Authorization`, `Cookie` и иные чувствительные значения должны сохраняться как есть.

### 4.6. Декодирование body

Сервис должен пытаться представить body в читаемом виде:

- Если body содержит JSON, его нужно разобрать и сохранить в форматированном виде.
- Если body содержит `application/x-www-form-urlencoded`, нужно декодировать параметры.
- Если body содержит `multipart/form-data`, нужно сохранить структуру частей, имена полей, имена файлов, заголовки частей и содержимое частей в полном объеме.
- Если body содержит текст в иной кодировке, нужно попытаться декодировать его в текстовом виде.
- Если body бинарное или не поддается корректному декодированию, нужно сохранить:
  - исходные данные в безопасном представлении;
  - явный признак того, что тело является бинарным или нераспознанным.

## 5. Требования к логам

### 5.1. Общие требования

- Для каждого запроса должны создаваться два отдельных файла:
  - один `txt`;
  - один `json`.
- Один запрос соответствует ровно одной записи в одном `txt`-файле и одному JSON-объекту в одном `json`-файле.
- JSON-файл должен содержать один JSON-объект.

### 5.2. Формат TXT

TXT-файл должен быть удобен для ручного чтения и содержать все сведения о запросе в развернутом виде:
- метаданные запроса;
- заголовки;
- исходное body;
- декодированное представление body;
- признаки типа содержимого и результата декодирования.

### 5.3. Формат JSON

JSON-файл должен содержать структурированный JSON-объект со всеми данными запроса, пригодный для автоматического анализа.

### 5.4. Именование файлов логов

Имена файлов логов должны строиться по шаблону:
- `timestamp + request_id`

Допустимый пример:
- `2026-05-25T12-34-56.123456Z_9f2e6c7a.txt`
- `2026-05-25T12-34-56.123456Z_9f2e6c7a.json`

Для одного запроса `txt` и `json` должны иметь одинаковую базовую часть имени.

## 6. Требования к сертификатам и TLS

### 6.1. Базовый подход

Нужно использовать собственный локальный центр сертификации `CA`, созданный на стороне Ubuntu.

Схема должна быть следующей:
- создается корневой сертификат `CA`;
- от него выпускается серверный сертификат;
- корневой сертификат `CA` импортируется в доверенные корневые центры сертификации на Windows-клиенте.

Этот вариант выбран как более надежный и удобный, чем единичный самоподписанный серверный сертификат, потому что:
- можно перевыпускать серверные сертификаты без повторной настройки доверия;
- можно использовать тот же `CA` для других тестовых серверов;
- Windows-клиент будет доверять именно корню доверия, а не одному конкретному сертификату.

### 6.2. Требования к серверному сертификату

Серверный сертификат должен поддерживать оба варианта обращения клиента:
- по DNS-имени;
- по IP-адресу.

Поэтому в `SAN` должны быть предусмотрены:
- `DNS`-записи;
- `IP`-записи.

### 6.3. Что должно войти в результат работ

Нужны готовые команды для:
- выпуска корневого `CA` на Ubuntu;
- выпуска серверного ключа и серверного сертификата на Ubuntu;
- настройки `SAN` для DNS-имени и IP-адреса;
- запуска Python-сервиса с сертификатом и ключом;
- импорта корневого `CA` в Windows;
- проверки того, что Windows доверяет серверу.

## 7. Инструкции, которые должны быть подготовлены

### 7.1. Ubuntu

Должна быть инструкция с готовыми командами:
- создания приватного ключа корневого `CA`;
- создания корневого сертификата `CA`;
- создания приватного ключа сервера;
- создания `CSR`;
- подписи серверного сертификата корневым `CA`;
- добавления `SAN` с `DNS` и `IP`;
- запуска сервиса из консоли.

### 7.2. Windows

Должна быть инструкция с готовыми командами или штатными шагами:
- импорта корневого сертификата в `Trusted Root Certification Authorities`;
- проверки корректности доверия к серверному сертификату;
- проверки, что HTTPS-запросы к сервису выполняются без ошибки недоверенного сертификата.

## 8. Нефункциональные требования

- Решение должно быть простым для запуска и сопровождения.
- Код должен быть пригоден для отладочного использования без сложной инфраструктуры.
- Логи должны быть пригодны и для ручного чтения, и для машинной обработки.
- Сервис должен работать стабильно в сценарии приема произвольных отладочных запросов.

## 9. Критерии приемки

Решение считается готовым, если одновременно выполняются следующие условия:

- сервис доступен по `HTTPS` на порту `51401`;
- сервис принимает запросы на любых путях;
- сервис принимает любые HTTP-методы;
- сервис принимает запросы от любого доступного клиента;
- на любой корректно принятый запрос сервис отвечает `200 OK`;
- каждый запрос логируется полностью без маскирования и без ограничения размера;
- для каждого запроса создаются отдельные файлы `txt` и `json`;
- JSON-файл содержит один JSON-объект;
- имена файлов формируются по шаблону `timestamp + request_id`;
- body сохраняется в исходном и, где возможно, в декодированном читабельном виде;
- сертификаты могут быть выпущены на Ubuntu по подготовленной инструкции;
- Windows-клиент доверяет серверу после импорта корневого `CA`;
- серверный сертификат работает как при обращении по DNS-имени, так и по IP-адресу.

В результате должны быть подготовлены:
- исходный код Python-сервиса;
- инструкция по запуску сервиса из консоли;
- инструкция с готовыми командами по выпуску сертификатов на Ubuntu;
- инструкция с готовыми командами или штатными шагами по импорту доверия на Windows;
- примеры логов `txt` и `json`.
